Response caching

💸

Cost & token analytics

🔄

Multi-provider routing

Native OpenAI + Anthropic. Model-based routing, automatic failover, retry with backoff.

🛡️

Circuit breaker

Per-provider breaker (Closed → Open → Half-open). Protects quota and avoids cascade failures.

Monthly budgets

Real-time dashboard

Next.js console: overview, spend charts, per-key breakdown, live req/min, key management.

Readiness probe

GET /readyz pings Redis + Postgres with a 2 s deadline — Kubernetes/ECS native.

How a request flows

Every call through POST /v1/chat/completions passes this pipeline.

1

Auth — virtual key lookup

Bearer token is SHA-256 hashed and matched against the api_keys table. Unknown key → 401.

2

Budget gate

Current month's spend (tracked in Redis) is compared to the key's monthly_budget_usd. Over budget → 429.

3

Rate limit

Redis token-bucket checks requests-per-minute for the key. Exceeded → 429 with Retry-After.

4

Cache lookup

Exact Redis match (SHA-256 of body). On hit, returns instantly with X-Cache: HIT. On miss, falls through to semantic cache (pgvector cosine similarity). Both paths are free of provider cost.

5

Provider routing + resilience

Model name selects provider (OpenAI or Anthropic). Request is retried up to 3× on transient failures (5xx / 429 / transport errors). If the provider trips the circuit breaker, failover activates the alternate provider.

6

Async logging

Tokens, cost, latency, model, cache status are written via a buffered channel + worker pool — never blocking the response path.

Quickstart

Up and running in under two minutes. Pick the path that fits — pre-built images need only Docker, no source code.

shell — recommended 
# 1. Grab the compose file + env template (no git clone needed)
curl -O https://raw.githubusercontent.com/sushil23harsana/ai-gateway/main/docker-compose.prod.yml
curl -o .env https://raw.githubusercontent.com/sushil23harsana/ai-gateway/main/.env.example
# Edit .env — add your OPENAI_API_KEY and/or ANTHROPIC_API_KEY

# 2. Pull the pre-built images and start (gateway :8080, dashboard :3000)
docker compose -f docker-compose.prod.yml up -d

# 3. Check health
curl http://localhost:8080/healthz
# → {"status":"ok"}

Images are published to ghcr.io/sushil23harsana/ai-gateway. Pin a version with GATEWAY_IMAGE=...:v1.0.0; the default tracks latest.

shell — for developers 
# 1. Clone and configure
git clone https://github.com/sushil23harsana/ai-gateway.git
cd ai-gateway
cp .env.example .env
# Edit .env — add your OPENAI_API_KEY and/or ANTHROPIC_API_KEY

# 2. Build locally and start the full stack
docker compose up -d --build

# 3. Check health
curl http://localhost:8080/healthz
# → {"status":"ok"}

Then open the dashboard at http://localhost:3000.

Mint a virtual key from the dashboard (API Keys page) or via the admin API:

curl -X POST http://localhost:8080/admin/keys \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"my-app","rpm_limit":60,"monthly_budget_usd":10}'

The response includes the raw key — save it, it is shown only once.

Using the gateway directly

The gateway is a drop-in replacement for the OpenAI API — same endpoint shape, virtual key as the Bearer token.

shell 
# Route to OpenAI (default)
curl http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer sk-gw-your-virtual-key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o-mini",
    "messages": [{"role": "user", "content": "Hello!"}]
  }'

# Route to Anthropic — just change the model name
curl http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer sk-gw-your-virtual-key" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "claude-sonnet-4-6",
    "messages": [{"role": "user", "content": "Hello from Anthropic!"}]
  }'
python 
from openai import OpenAI

# Point the official OpenAI SDK at Janus — no other change needed
client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="sk-gw-your-virtual-key",
)

# OpenAI model
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Explain backpressure in 2 sentences."}],
)
print(response.choices[0].message.content)

# Anthropic model — same client, same call, just a different model name
response = client.chat.completions.create(
    model="claude-sonnet-4-6",
    messages=[{"role": "user", "content": "Explain backpressure in 2 sentences."}],
)
print(response.choices[0].message.content)
python 
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="sk-gw-your-virtual-key",
)

# Streaming works exactly like the standard SDK — tokens are accounted
# on the gateway side even in stream mode.
with client.chat.completions.stream(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Write a haiku about distributed systems."}],
) as stream:
    for text in stream.text_stream():
        print(text, end="", flush=True)

Using with LangChain

Because Janus speaks the OpenAI chat completions format, any LangChain integration that accepts a base_url works out of the box — no custom provider needed.

python 
# pip install langchain-openai
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# Point LangChain at Janus — openai_api_base is the key
llm = ChatOpenAI(
    model="gpt-4o-mini",
    openai_api_key="sk-gw-your-virtual-key",
    openai_api_base="http://localhost:8080/v1",
)

response = llm.invoke([HumanMessage(content="What is a circuit breaker in software?")])
print(response.content)

# Switch to Anthropic — same LangChain code, just change model name
llm_claude = ChatOpenAI(
    model="claude-sonnet-4-6",
    openai_api_key="sk-gw-your-virtual-key",
    openai_api_base="http://localhost:8080/v1",
)
response = llm_claude.invoke([HumanMessage(content="What is a circuit breaker in software?")])
print(response.content)
python 
# pip install langchain langchain-openai
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

llm = ChatOpenAI(
    model="gpt-4o-mini",
    openai_api_key="sk-gw-your-virtual-key",
    openai_api_base="http://localhost:8080/v1",
    temperature=0,                  # deterministic → cache hits are likely
    streaming=True,
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a concise technical writer."),
    ("human",  "Summarise {topic} in one paragraph."),
])

chain = prompt | llm | StrOutputParser()

# First call hits OpenAI; second call (same topic) returns from Janus cache
print(chain.invoke({"topic": "Redis Streams"}))
print(chain.invoke({"topic": "Redis Streams"}))  # → X-Cache: HIT, 0 cost
python 
# pip install langchain langchain-openai
from langchain_openai import ChatOpenAI
from langchain.agents import AgentExecutor, create_openai_tools_agent
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.tools import tool

# Tool use / function calling works too
@tool
def get_word_count(text: str) -> int:
    """Returns the number of words in the text."""
    return len(text.split())

llm = ChatOpenAI(
    model="gpt-4o",
    openai_api_key="sk-gw-your-virtual-key",
    openai_api_base="http://localhost:8080/v1",
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human",  "{input}"),
    MessagesPlaceholder(variable_name="agent_scratchpad"),
])

agent  = create_openai_tools_agent(llm, [get_word_count], prompt)
runner = AgentExecutor(agent=agent, tools=[get_word_count], verbose=True)

result = runner.invoke({"input": "How many words are in 'The quick brown fox'?"})
print(result["output"])
Every agent step — including tool-call turns — flows through Janus and appears in your dashboard analytics. Budget and rate limits apply per-key across all agent invocations.
typescript 
// npm install @langchain/openai @langchain/core langchain
import { ChatOpenAI } from "@langchain/openai";
import { HumanMessage } from "@langchain/core/messages";

// Point LangChain JS at Janus
const llm = new ChatOpenAI({
  modelName:    "gpt-4o-mini",
  openAIApiKey: "sk-gw-your-virtual-key",
  configuration: {
    baseURL: "http://localhost:8080/v1",
  },
});

const response = await llm.invoke([
  new HumanMessage("What is the capital of France?"),
]);
console.log(response.content);

// Streaming
const stream = await llm.stream([
  new HumanMessage("Write a haiku about caching."),
]);
for await (const chunk of stream) {
  process.stdout.write(chunk.content as string);
}
typescript — LCEL chain 
import { ChatOpenAI } from "@langchain/openai";
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { StringOutputParser } from "@langchain/core/output_parsers";

const model = new ChatOpenAI({
  modelName:    "gpt-4o-mini",
  openAIApiKey: "sk-gw-your-virtual-key",
  configuration: { baseURL: "http://localhost:8080/v1" },
});

const prompt = ChatPromptTemplate.fromMessages([
  ["system", "You are a concise technical writer."],
  ["human",  "Summarise {topic} in one paragraph."],
]);

const chain = prompt.pipe(model).pipe(new StringOutputParser());
console.log(await chain.invoke({ topic: "Redis Streams" }));
Why this works: Janus exposes the exact same POST /v1/chat/completions shape as the OpenAI API. Any client that accepts a custom base_url — openai-python, openai-node, LangChain, LlamaIndex, AutoGen, CrewAI — works without modification beyond pointing at http://<gateway-host>:8080/v1.

Key configuration

All config is via environment variables (12-factor). Copy .env.example to .env and fill in what you need.

Variable Default Description
OPENAI_API_KEY—Your OpenAI secret key (server-side only, never exposed)
ANTHROPIC_API_KEY—Your Anthropic secret key (server-side only, never exposed)
ADMIN_TOKENchange-meProtects /admin/* endpoints. Set a strong random value for production.
PORT8080Gateway listen port
DEFAULT_PROVIDERopenaiWhich provider handles requests when not determinable by model name
FAILOVER_ENABLEDtrueAuto-failover to the other provider on 5xx / timeout
CACHE_TTL_SECONDS3600Exact-match cache TTL
BUDGET_ENFORCEDtrueEnforce per-key monthly budget caps
RETRY_MAX_ATTEMPTS3Max retries on transient upstream failures
BREAKER_ENABLEDtrueEnable per-provider circuit breaker
SEMANTIC_CACHE_ENABLEDfalsepgvector semantic cache (requires OPENAI_API_KEY for embeddings)
Production checklist: set a strong ADMIN_TOKEN (openssl rand -hex 32), bind ports to loopback only (default in Docker Compose), and put the dashboard behind a reverse proxy with TLS. See docs/security.md.
